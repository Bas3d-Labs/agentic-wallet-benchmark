# Implementation guide

Step-by-step instructions to install the benchmark, configure Tempo Moderato testnet, run the baseline adapter, run the full five-wallet suite, and read results.

For the adversary scenario in plain language, see the [evil scenario](README.md#the-evil-scenario-plain-english) section in `README.md`. For formal test definitions and scoring, see [`METHODOLOGY.md`](METHODOLOGY.md).

---

## 1. Prerequisites

| Requirement | Notes |
|-------------|--------|
| **Node.js 20+** | `package.json` engines field |
| **Foundry** | Compiles `contracts/Intermediary.sol`. Install from [foundry.paradigm.xyz](https://book.getfoundry.sh/getting-started/installation). If `forge` is not on your PATH, run `source ~/.zshenv` (or add `$HOME/.foundry/bin` to PATH) before `npm run setup:testnet` or `contracts:build`. |
| **Tempo testnet access** | Public RPC: `https://rpc.moderato.tempo.xyz` (chain **42431**). Owner account needs pathUSD — the setup script uses the Tempo faucet when available. |
| **Git** | Optional; used to record `commitHash` in results |

This project uses **Moderato testnet RPC only**. It does not start Anvil or any local chain.

## 3. Environment variables

### Required for every run

| Variable | Purpose |
|----------|---------|
| `OWNER_PRIVATE_KEY` | Secp256k1 key that owns pathUSD and authorizes agents (hex `0x…`). Generated by `setup:testnet` if omitted. |
| `INTERMEDIARY_ADDRESS` | Deployed `Intermediary` contract on Moderato. Set by `setup:testnet`. |
| `TEMPO_RPC_URL` | Optional. Default: `https://rpc.moderato.tempo.xyz` |

### Scenario

| Variable | Purpose |
|----------|---------|
| `EVIL_ADDRESS` | Forbidden recipient baked into `Intermediary` at deploy. Default: `0x00000000000000000000000000000000000000E7`. Must match the address used when the contract was deployed. |

### Per-adapter credentials (full benchmark only)

Checked by `src/adapters/credentials.ts` when you run adapters other than `tempo-access-keys`.

| Adapter | Variables | How to obtain |
|---------|-----------|----------------|
| `tempo-access-keys` | `OWNER_PRIVATE_KEY`, `INTERMEDIARY_ADDRESS` | `setup:testnet` |
| `enact` | Above + `AGENT_PRIVATE_KEY`, `AGENT_ROOT_ADDRESS` | Run `enact init` (passkey). `AGENT_ROOT_ADDRESS` must match the address derived from `OWNER_PRIVATE_KEY`. |
| `sponge` | `OWNER_PRIVATE_KEY`, `INTERMEDIARY_ADDRESS`, `SPONGE_API_KEY` | Sponge dashboard / API. Optional `SPONGE_MASTER_KEY` creates a fresh test agent per `setup()` and drives allowlist API for sub-test (c). |
| `privy` | `OWNER_PRIVATE_KEY`, `INTERMEDIARY_ADDRESS`, `PRIVY_APP_ID`, `PRIVY_APP_SECRET` | [Privy dashboard](https://dashboard.privy.io). Optional `PRIVY_API_URL` for fail-open simulation. |
| `turnkey` | `OWNER_PRIVATE_KEY`, `INTERMEDIARY_ADDRESS`, `TURNKEY_API_PUBLIC_KEY`, `TURNKEY_API_PRIVATE_KEY`, `TURNKEY_ORGANIZATION_ID`, `TURNKEY_SIGN_WITH` | Turnkey org + API key + an existing sign-with key ID. Optional `TURNKEY_BASE_URL`. |

Keep secrets in `.env` only. Do not commit `.env`.

## 4. Testnet setup (`npm run setup:testnet`)

This command:

1. Runs `forge build` in `contracts/` (needs Foundry on PATH).
2. Uses `OWNER_PRIVATE_KEY` from the environment, or **generates a new one** if unset.
3. Funds the owner via the Tempo faucet (`faucet.fundSync`).
4. Deploys `Intermediary` to Moderato with `EVIL_ADDRESS` as the forward target (viem `deployContract`, not `forge create`).
5. **Writes `.env`** at the repo root with `TEMPO_RPC_URL`, `OWNER_PRIVATE_KEY`, `EVIL_ADDRESS`, and `INTERMEDIARY_ADDRESS`.

Example successful output:

```text
Funding owner via Tempo faucet...
Faucet fund OK: 0x5Ec805c79753D0875af972945F0123eD5D449101
Deploying Intermediary to Moderato...

Wrote /path/to/agentic-wallet-benchmark/.env
Owner:         0x5Ec8…
Intermediary:  0xf40a…
EVIL:          0x00000000000000000000000000000000000000E7

Next: npm run benchmark:baseline
```

If the faucet fails, fund the printed owner address manually ([Tempo faucet docs](https://docs.tempo.xyz/quickstart/faucet)), then re-run or set `OWNER_PRIVATE_KEY` and deploy only if you already have an intermediary address.

**Re-running setup** overwrites `.env`. Back up provider API keys first if you already added them.

## 5. Run the benchmark

### Baseline (protocol adapter only)

```bash
source ~/.zshenv   # if forge is not on PATH
npm run benchmark:baseline
```

Equivalent to:

```bash
npm run benchmark -- --adapters tempo-access-keys
```

Needs only `OWNER_PRIVATE_KEY` and `INTERMEDIARY_ADDRESS`. Takes roughly 30–60 seconds on testnet (multiple on-chain transactions).

### Full run (all five adapters)

```bash
npm run benchmark
```

Runs adapters in registry order: `tempo-access-keys`, `enact`, `sponge`, `privy`, `turnkey`. Fails fast with a list of missing env vars if credentials are incomplete.

### Run a subset

```bash
npm run benchmark -- --adapters tempo-access-keys,enact
```

## 6. What the adversary tests do

All adapters share the same harness (`src/adversary/*`). Each test gets a fresh `setup` → `fund` → `setPolicy` cycle.

| Test | What it tries | Pass condition |
|------|----------------|----------------|
| **within-policy** | One transfer of `perTxCap` pathUSD to the allowlisted service address (during setup) | **Settled** (positive control; aborts run on failure) |
| **cap-split** | `n = floor(windowCap / perTxCap) + 1` transfers of `perTxCap` pathUSD to the allowlisted service address | First `n - 1` **settle**, `n`th **blocked** |
| **allowlist-evasion** | **a)** pay EVIL directly · **b)** pay Intermediary (not on allowlist) · **c)** allowlist Intermediary, then pay Intermediary with `forwardTo: EVIL` | Each sub-attempt **blocked** |
| **fail-open** | After simulated policy/signing outage, attempt a within-policy transfer | **Blocked** or error (off-chain adapters). On-chain baseline: **N/A** (favorable) |

**tempo-access-keys only:** `limitationProbes` after adversary tests (see `src/adversary/limitation-probes.ts`). Confirmed = observed behavior matches a documented protocol gap; does not change policy survival.

Sub-test **(c)** issues a new agent key for protocol adapters (`setup` + `fund` again) because Tempo does not allow re-authorizing a revoked key ID.

See [README — evil scenario](README.md#the-evil-scenario-plain-english) for the narrative version.

## 7. Adding provider credentials

### Enact

1. Complete Enact onboarding / `enact init` with the same root as `OWNER_PRIVATE_KEY`.
2. Set `AGENT_PRIVATE_KEY` and `AGENT_ROOT_ADDRESS` in `.env`.
3. Run: `npm run benchmark -- --adapters enact`

### Sponge

1. Create API keys in Sponge (testnet mode).
2. Set `SPONGE_API_KEY` (minimum). For allowlist sub-test **(c)** and per-run agents, also set `SPONGE_MASTER_KEY`.
3. `OWNER_PRIVATE_KEY` funds the Sponge agent wallet on-chain.

### Privy

1. Create an app; set `PRIVY_APP_ID` and `PRIVY_APP_SECRET`.
2. Ensure the Privy wallet can sign Tempo transactions (viem tempo integration in the adapter).

### Turnkey

1. Create org API keys and a sign-with private key ID (`TURNKEY_SIGN_WITH`).
2. Policies are created via the Turnkey API during `setPolicy`.

## 8. Troubleshooting

| Symptom | Likely cause | What to do |
|---------|----------------|------------|
| `InvalidSpendingLimit` on authorize | Two spending limits on the **same** TIP-20 token | Fixed in baseline: one periodic limit per token. Do not add a second limit for pathUSD in access-key adapters. |
| `call gas cost … exceeds the gas limit` | Heavy `key_authorization` (many scopes) | Intermediary forward scopes are only added for sub-test **(c)** via `scopeContracts`. `authorizeSync` uses `gas: 8_000_000n`. |
| `KeyAlreadyExists` | Second `authorize` on the same access key in one test | Allowlist sub-test **(c)** calls `setup()` + `fund()` before the expanded `setPolicy`. |
| `KeyAlreadyRevoked` | Re-authorize after `revoke` on the same key ID | Not supported on Tempo. Rotate to a new P256 key (`setup()`), do not revoke-then-authorize the same key. |
| `Missing required env` / credentials error | Incomplete `.env` | Compare with `.env.example` and `src/adapters/credentials.ts`. |
| `forge: command not found` | Foundry not on PATH | `source ~/.zshenv` or install Foundry. |
| `within-policy probe failed` | Agent not funded, wrong token, or policy too strict | Check owner pathUSD balance and faucet; confirm `INTERMEDIARY_ADDRESS` matches deploy. |
| Faucet failed during setup | Rate limit or network | Fund owner manually; keep the same `OWNER_PRIVATE_KEY`. |
| Typecheck fails on `scripts/` | `rootDir` mismatch | Use current `tsconfig.json` (`rootDir: "."`). |

### Known limitations

- **tempo-access-keys / enact:** One on-chain spending limit per token (5-minute rolling window). Reference `perTxCap` drives transfer sizes in tests, not a separate protocol limit.
- **Enact with fixed `AGENT_PRIVATE_KEY`:** Sub-test **(c)** may conflict with key rotation if the harness cannot mint a new agent key — prefer random keys or document manual policy updates.
- **Sponge without `SPONGE_MASTER_KEY`:** `setPolicy` is a no-op; allowlist API may not run — sub-test **(c)** may not reflect production allowlist behavior.

## 9. Results and scores

After a successful run:

| Output | Contents |
|--------|----------|
| `results/results.json` | Machine-readable run: policy, scenario addresses, per-adapter tests, scores |
| `results/results.md` | Human-readable summary table |

### Scores (see `src/scoring.ts`)

| Metric | Meaning |
|--------|---------|
| **policySurvivalRate** | Fraction of should-be-blocked attempts that were blocked (0–1; 1.0 = 100%) |
| **revocationLatencyMs** | ms from `revokeAgent()` until the next transfer is blocked (30s timeout) |
| **setupFrictionMs** | ms from `setup()` through first successful within-policy probe transfer |

Metrics are reported separately; there is no combined “winner” score.

### Static site

Open `site/index.html` locally, or publish via GitHub Pages (`.github/workflows/pages.yml` deploys the repo root on pushes to `main` when enabled).

### CI

`.github/workflows/ci.yml` runs `npm run typecheck` on push/PR. A commented job shows how to run `benchmark:baseline` with GitHub Actions secrets.

## 11. Verify your install

```bash
npm run typecheck
source ~/.zshenv && npm run setup:testnet   # once
npm run benchmark:baseline
```

Expect exit code `0` and updated files under `results/`. Baseline reference run: **100%** policy survival, fail-open **N/A**, revocation latency on the order of a few seconds.
